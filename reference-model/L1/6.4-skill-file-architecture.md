## 1. Purpose

Skill files are the core encoding format of the Plektis. They are structured `.md` files that encode senior practitioner expertise as machine-readable instructions that AI agents can execute. The skill file pattern is the foundational innovation of the protocol: it makes tacit professional knowledge portable, versionable, and executable across agents and platforms.

This specification defines:

- The canonical structure every skill file must follow
- The YAML frontmatter interface, conformant with the AAIF Agent Skills standard
- The two-type classification system
- Security constraints and trust levels
- Naming and versioning conventions
- The skill file lifecycle from creation through evolution to deprecation

This specification governs all files distributed through the Layer 2 SDK and all files used in Layer 4 first-party products and Layer 5 integrated deployments.

### 1.1 Relationship to the AAIF Agent Skills Standard

The Plektis Skill File Architecture is a **domain-specific extension** of the Agent Skills standard published by the Agentic AI Foundation (AAIF) at agentskills.io. The AAIF standard, launched December 2025 under the Linux Foundation and co-founded by Anthropic, OpenAI, and Block, defines the universal format for SKILL.md files across the agentic AI ecosystem.

Plektis skill files are fully AAIF-conformant: they use the standard top-level frontmatter fields (`name`, `description`, `license`, `compatibility`, `allowed-tools`) and place all domain-specific extensions under the `metadata:` map as the AAIF spec recommends. This means Plektis skill files are interoperable with any AAIF-compatible agent while carrying the additional metadata that Plektis SDK tooling requires.

The AAIF standard defines what a skill file IS. Plektis defines what a skill file knows about business architecture, strategy execution, and professional advisory practice.

## 2. Classification: Two Types

Skill files serve one of two functions. The function determines the structural template.

### 2.1 Domain Intelligence

A domain intelligence file provides **persistent context**. It tells the agent "who you are and what you know." It is loaded as background, not invoked with arguments.

**Characteristics:**

- Changes how the agent thinks, not what the agent does next
- Contains substantial domain knowledge: frameworks, taxonomies, signal models, scoring methodologies
- Defines quality standards and anti-patterns for a knowledge domain
- Does not prescribe a step-by-step workflow
- May be loaded alongside other skill files to compound context

**Examples from the existing portfolio:**

| File | Domain | Purpose |
|---|---|---|
| (Domain-specific skill) | (Practitioner's domain) | Signal taxonomies, scoring methodologies, domain frameworks |
| (Domain-specific skill) | (Practitioner's domain) | Domain-specific knowledge, analytical frameworks, quality standards |
| Business Architecture Foundations | Business architecture | Capability mapping, value streams, BMM, PPIT, roadmapping |

### 2.2 Operational Playbook

An operational playbook file provides **executable instructions**. It tells the agent "what to do right now." It is invoked with parameters, follows a workflow, and produces defined outputs.

**Characteristics:**

- Prescribes a sequence: do this, then this, then this
- Produces a defined output artifact
- May be triggered with arguments that route to different paths
- May delegate to reference files for deep content
- Includes routing logic (decision trees, engagement type selection, conditional branching)

**Examples** (illustrative of a full deployment configuration. These operational playbooks are commercial-layer components, not part of the public SDK):

| File | Domain | Purpose |
|---|---|---|
| Engagement Playbook | Business architecture | Engagement type routing (full assessment, rapid diagnostic, TOM, strategy-to-execution) |
| Capability Assessment | Business architecture | Structured maturity assessment with scoring dimensions and gap analysis |
| TOM Design | Business architecture | Target Operating Model design with PPIT dimensional analysis |

### 2.3 Classification Criteria

When creating a new skill file, apply this test:

1. **Is it loaded as background context?** If yes: domain intelligence.
2. **Is it invoked to produce a specific output?** If yes: operational playbook.
3. **Does it have a step-by-step workflow?** If yes: operational playbook.
4. **Does it define what NOT to do more than what TO do?** If yes: domain intelligence.

If the answer is ambiguous, default to domain intelligence. A file can always be refactored into a playbook later if a workflow emerges through practice.

## 3. YAML Frontmatter Specification

Every skill file must have a YAML frontmatter block at the top. The frontmatter is structured in three tiers to ensure AAIF compatibility while carrying Plektis-specific metadata.

### 3.1 Tier 1: AAIF Standard Fields (Top-Level)

These fields conform to the Agent Skills specification (agentskills.io/specification). They are parsed by any AAIF-compatible agent.

```yaml
---
name: string
# Kebab-case unique identifier. Max 64 characters.
# Lowercase letters, numbers, and hyphens only.
# Must match the parent directory name (AAIF requirement).
# Must not start/end with a hyphen or contain consecutive hyphens.
# Examples: "succession-intelligence", "capability-assessment"

description: string
# Max 1024 characters. Must answer: "When should an agent load this file?"
# This is the primary trigger signal for both human selection and AI auto-loading.
# Should include specific keywords that help agents identify relevant tasks.

license: string
# License name or reference to a bundled license file.
# For Plektis SDK skills: "CC BY-SA 4.0"

compatibility: string
# Max 500 characters. Environment requirements.
# Indicate intended platforms, system packages, or network access needs.
# Example: "Designed for business architecture engagements. Works with Claude Code,
# Gemini, and other AAIF-compatible agents."

allowed-tools: string
# Space-separated list of pre-approved tools. AAIF format.
# Example: "Read Grep Glob Bash(sqlite3:*)"
# Omit if no tool restrictions apply.
# Note: Claude Code historically used comma-separated format.
# The AAIF space-separated format is the standard going forward.

metadata: map
# Arbitrary key-value mapping for additional properties.
# Plektis-specific fields live here (see Tier 3 below).
---
```

**Required fields:** `name` and `description` are mandatory per the AAIF spec and this specification. All other Tier 1 fields are optional per AAIF but `license` is mandatory for Plektis SDK distribution.

### 3.2 Tier 2: Platform Runtime Fields (Top-Level)

These fields are parsed by specific AI platforms (currently Claude Code). They are not part of the AAIF standard but do not conflict with it - unknown top-level fields are silently ignored by AAIF-compatible agents.

```yaml
---
user-invocable: boolean
# Default: false. If true, practitioners can trigger directly via /name command.
# Claude Code-specific. Only meaningful for operational playbooks.

argument-hint: string
# Describes expected arguments when invoked.
# Only meaningful when user-invocable: true.
# The body can reference $ARGUMENTS as a placeholder for substitution.
# Example: "<scope: L1-name | 'all'> [dimensions: maturity | all]"

disable-model-invocation: boolean
# If true, prevents the agent from auto-loading this skill based on
# conversation context. Skill only activates via explicit /name command.
# Claude Code-specific. Default: false.
---
```

These fields are optional. Include only when the skill has platform-specific invocation requirements.

### 3.3 Tier 3: Plektis Extensions (Under `metadata:`)

These fields are specific to the Plektis ecosystem. Per the AAIF spec recommendation ("make your key names reasonably unique to avoid accidental conflicts"), all Plektis metadata keys are prefixed with `plektis-`.

AAIF-compatible agents that are not Plektis-aware will ignore these fields. Plektis SDK tooling reads them for catalog management, type classification, lifecycle tracking, and security enforcement.

```yaml
---
metadata:
  plektis-version: string
  # Semantic version: major.minor
  # Major: structural or interface changes
  # Minor: content refinements, additions, corrections
  # v0.x signals draft/untested state
  # Examples: "2.0", "1.0", "0.1"

  plektis-type: string
  # One of: "domain-intelligence" | "operational-playbook"
  # Determines which section template applies (see Section 4).

  plektis-domain: string | string[]
  # The knowledge domain, or a list of domains for a skill that spans several.
  # Use consistent vocabulary across the catalog.
  # Examples: "business-architecture"; or, for a multi-domain skill,
  # ["investment-portfolio", "m-and-a-advisory", "capital-allocation", "succession-planning"]

  plektis-author: string
  # Primary author name.

  plektis-status: string
  # One of: "draft" | "active" | "deprecated"
  # Tracks lifecycle state (see Section 6).

  plektis-trust-level: string
  # One of: "first-party" | "curated" | "community"
  # Declares the provenance of this skill file (see Section 4.3).

  # === Optional Plektis metadata ===

  plektis-senref-phases: string[]
  # Which phases of the machine's cognitive cycle this skill configures.
  # One or more of: "sense" | "model" | "decide" | "reflect"
  # See L0 reference model Section 3.2.1.

  plektis-confidence-profile: string
  # One of: "high" | "medium" | "low"
  # Signals how battle-tested this skill is.
  # high: refined through 3+ real engagements with iterative feedback
  # medium: tested in 1-2 engagements
  # low: draft, theoretical, or untested

  plektis-role: string
  # Which agent in a multi-agent pattern should load this.
  # One of: "architect" | "implementer" | "any"
  # Default: "any"

  plektis-requires: string[]
  # Skill file names that must be loaded as context for this skill to function.
  # Creates a dependency that SDK tooling can resolve.

  plektis-references: string[]
  # Paths to companion files, relative to the skill's root directory.
  # Used when the skill delegates deep content to external files.

  plektis-last-updated: string
  # ISO date: YYYY-MM-DD
---
```

**Required Plektis metadata:** `plektis-version`, `plektis-type`, `plektis-domain`, `plektis-author`, `plektis-status`, and `plektis-trust-level` are mandatory for all Plektis SDK skills.

### 3.4 Frontmatter Design Rationale

**Why three tiers:** The AAIF standard defines a minimal, universal schema. Platform-specific fields (Claude Code) sit alongside it without conflict. Plektis-specific fields go under `metadata:` per the AAIF recommendation. This structure means a single file is simultaneously valid AAIF, valid Claude Code, and valid Plektis - each system reads what it understands and ignores the rest.

**Why `plektis-` prefix:** The AAIF spec recommends making metadata key names "reasonably unique to avoid accidental conflicts." The prefix ensures no collision with other metadata consumers.

**Why `plektis-type` instead of a top-level `type`:** An earlier design placed `type` at the top level. Moving it under `metadata:` maintains AAIF conformance. The classification system is unchanged - only the YAML location moved.

**Why `license` is mandatory for Plektis:** The AAIF spec makes `license` optional. Plektis makes it mandatory for SDK distribution to ensure every skill file carries its terms. The standard Plektis license is CC BY-SA 4.0.

## 4. Section Templates

### 4.1 Domain Intelligence Template

Sections marked `[R]` are required. Sections marked `[O]` are optional.

```
---
[Tier 1: AAIF fields - name, description, license, compatibility]
[Tier 3: metadata with plektis-type: "domain-intelligence"]
---

# [Skill Name]                                                    [R]

## Purpose                                                         [R]
What this file encodes and how it calibrates agent outputs.
2-3 paragraphs. Must answer: "When this file is loaded,
what changes about the agent's behaviour?"

## Professional Context                                            [R]
Domain scope, application context, strategic positioning.
Situates the skill within a real operational environment.

## [Domain Knowledge Sections]                                     [R]
The substantive content. Named and structured per the domain.
No prescribed names. Use H2 for major sections, H3 for
subsections. Tables encouraged for structured data.

## Quality Standards                                               [R]
Analytical, communication, and ethical standards that govern
any output produced while this skill is active.

## Anti-Patterns                                                   [R]
What NOT to do. Named prohibitions, not vague guidelines.
The test: if someone could violate this rule accidentally,
it belongs here.

## Glossary                                                        [O]
Term-definition table for domain-specific vocabulary.
Recommended when the skill introduces 10+ domain-specific terms.

## Version History                                                 [O]
Table: Version | Date | Author | Summary
Required when the file has been revised (v1.1+).
Reads newest-to-oldest.
```

### 4.2 Operational Playbook Template

```
---
[Tier 1: AAIF fields - name, description, license, compatibility, allowed-tools]
[Tier 2: Claude Code fields - user-invocable, argument-hint, disable-model-invocation]
[Tier 3: metadata with plektis-type: "operational-playbook"]
---

# [Skill Name]                                                    [R]

[Role sentence: "You are a [role] performing [task]." One line.]   [R]

## Scope: $ARGUMENTS                                               [R*]
* Required when user-invocable: true with argument-hint.

## Routing / Decision Logic                                        [O]
Decision tree, routing table, or conditional branching.

## [Workflow Steps]                                                 [R]
The operational sequence. Each step specifies:
- Objective, Activities, Tools/skills, Output, Quality gates

## Output Specification                                            [R]
The expected output format. Use fenced code blocks for templates.

## Quality Checklist                                               [R]
Verification checklist before finalizing output.

## Practitioner Guidance                                           [O]
Tips, heuristics, and judgment calls. 3-7 bullet points.

## Version History                                                 [O]
Same convention as domain intelligence files.
```

### 4.3 Security Constraints

Every skill file operates within a trust boundary. These constraints apply regardless of skill type.

#### 4.3.1 Trust Levels

The `plektis-trust-level` metadata field declares the provenance of the skill file:

| Trust Level | Source | Verification | Runtime Behavior |
|---|---|---|---|
| **first-party** | Authored by the node's driver or their organization. | Verified by authorship and commit history. | Full execution within the node's authorized scope. |
| **curated** | Third party, reviewed through the Layer 3 marketplace quality gate. | Signed by the curator. | Execution with enhanced monitoring. |
| **community** | Downloaded from open repositories. No formal review. | Unverified. Use at own risk. | Agent flags to driver before any Tier 1 or Tier 2 escalation action. |

If `plektis-trust-level` is absent, the file is treated as `community` (highest risk).

#### 4.3.2 Prohibited Instructions

A skill file must NEVER contain instructions that:

1. **Exfiltrate data.** Send engagement data, client information, or node state to external endpoints not authorized by the driver.
2. **Bypass escalation triggers.** Override or circumvent the three-tier escalation framework (L0 Section 3.3.4). No skill file may instruct the agent to skip human authorization for Tier 1 or Tier 2 actions.
3. **Modify other skill files.** A skill may propose updates to itself through the Reflect phase, but must NEVER directly modify another skill file's content.
4. **Escalate its own permissions.** A skill may not instruct the agent to grant itself access to tools beyond its `allowed-tools` declaration.
5. **Persist state outside the defined persistence model.** No writing to system files, environment variables, or external storage without explicit driver authorization (L0 Section 3.3.5).
6. **Impersonate the driver.** No communications or actions representing themselves as coming from the driver unless explicitly authorized through a Tier 1 escalation.

#### 4.3.3 Context Isolation

- Client data from Engagement A must not be accessible during Engagement B unless the driver explicitly authorizes cross-engagement sharing.
- A skill file's Quality Standards and Anti-Patterns apply only within its declared domain.
- In multi-agent configurations, each agent operates within its own trust boundary.

#### 4.3.4 Provenance Requirements (SDK and Marketplace)

For any skill file distributed through Layer 2 or Layer 3:

- `plektis-author` must be declared. Anonymous skill files are not accepted.
- Version History must be maintained for any file revised past v1.0.
- Git commit history is the source of truth for authorship provenance.
- Marketplace (Layer 3) skills may require cryptographic signing.

## 5. Directory Structure and Naming

### 5.1 AAIF-Conformant Directory Structure

Per the AAIF spec, each skill lives in its own directory with the `name` field matching the directory name:

```
skill-name/
├── SKILL.md          # Required: metadata + instructions
├── scripts/          # Optional: executable code
├── references/       # Optional: companion documentation
└── assets/           # Optional: templates, resources
```

This is the canonical structure for the Plektis GitHub repo, the Layer 2 SDK, and any AAIF-compatible distribution.

### 5.2 Claude Code Platform Convention

When deployed inside `.claude/skills/`, the same structure applies:

```
.claude/skills/skill-name/SKILL.md
```

The directory name serves as the identifier. The version lives in the frontmatter under `metadata.plektis-version`.

### 5.3 Versioning Semantics

| Version Pattern | Meaning | Examples |
|---|---|---|
| v0.x | Draft. Not yet tested in a real engagement. | v0.1 (initial draft), v0.3 (third iteration) |
| vX.0 | Major release. Structural or interface changes. | v1.0 (first production release), v2.0 (major restructure) |
| vX.Y | Minor release. Content additions, refinements, corrections. | v1.1 (added industry patterns), v3.2 (15th editorial rule) |

**When to bump major:** Restructuring frontmatter tiers. Changing the type classification. Restructuring sections. Changing the output specification.

**When to bump minor:** Adding new domain knowledge. Refining anti-patterns. Expanding glossary. Adding editorial rules. Fixing errors.

### 5.4 Version History Table

```markdown
## Version History

| Version | Date | Author | Summary |
|---------|------|--------|---------|
| 2.0 | 2026-04-13 | Krish Jogia | AAIF-conformant frontmatter. Restructured metadata. |
| 1.0 | 2026-04-12 | Krish Jogia | Initial Plektis migration. |
```

Newest entry first. The Summary column is specific about what changed.

## 6. Skill File Lifecycle

A skill file moves through three states, tracked by the `plektis-status` metadata field:

### 6.1 Draft

```yaml
metadata:
  plektis-status: draft
  plektis-confidence-profile: low
```

The file exists but has not been tested in a real engagement. Draft files may be distributed in the SDK with the explicit understanding that they are experimental.

**Transition to active:** The file is used in at least one real engagement. The practitioner confirms that the agent behavior it produces is correct and useful.

### 6.2 Active

```yaml
metadata:
  plektis-status: active
  plektis-confidence-profile: medium  # or high
```

The file has been tested and is in use. Active files are the primary content of the SDK.

**Confidence escalation:** A file moves from `medium` to `high` confidence when it has been refined through 3+ engagements with iterative feedback.

### 6.3 Deprecated

```yaml
metadata:
  plektis-status: deprecated
```

Superseded or no longer maintained. Remains in the SDK for reference.

### 6.4 The Evolution Protocol

Skill files improve through a specific feedback loop (L0 reference model Section 3.3.6):

1. **The skill is loaded into an engagement.**
2. **The driver observes failures or refinement opportunities.**
3. **The driver corrects in-session.**
4. **The correction is codified in the skill file.** Version bumped.
5. **The updated skill is available for the next engagement.**

This is how a mature skill file evolves through iterative engagement feedback - each correction discovered in practice becomes a formal rule. The process is the same for every skill file in the protocol.

## 7. Alignment with L0 Operating Principles

| L0 Principle | How This Specification Implements It |
|---|---|
| 1. The driver decides. The machine prepares | The `plektis-role` metadata field declares which agent loads the skill. Escalation triggers from L0 Section 3.3.4 are enforced via the prohibited instructions in Section 4.3.2. |
| 2. Iterative over waterfall | The versioning system and evolution protocol (Section 6.4) make iteration visible. |
| 3. Decoupled over monolithic | The `plektis-requires` and `plektis-references` fields enable composition without monolithic files. |
| 4. Confidence-aware throughout | The `plektis-confidence-profile` metadata field. The security constraints (Section 4.3) enforce trust-level awareness. |
| 5. Graph-native over tabular | The `plektis-requires`/`plektis-references` fields create a dependency graph between skills. |
| 6. Agentic over procedural | Domain intelligence files calibrate behavior without prescribing procedures. Playbooks include Practitioner Guidance for judgment calls. |
| 7. Transferable by design | AAIF conformance ensures skills are portable across agents and platforms. The `plektis-trust-level` field makes provenance explicit. |

### SenRef Cognitive Cycle Integration

The `plektis-senref-phases` metadata field maps each skill file to the machine's cognitive cycle. Domain intelligence files calibrate specific phases (e.g., a domain intelligence skill configures Sense). Operational playbooks execute specific phases (e.g., Capability Assessment executes Decide). SDK tooling can use this field to ensure phase coverage across a domain's skill set.

## 8. Progressive Disclosure (AAIF Alignment)

Following the AAIF spec, skill files are structured for efficient use of context:

1. **Metadata (~100 tokens):** The `name` and `description` fields are loaded at startup for all skills.
2. **Instructions (<5000 tokens recommended):** The full SKILL.md body is loaded when the skill is activated.
3. **Resources (as needed):** Files in `scripts/`, `references/`, or `assets/` are loaded only when required.

Keep the main SKILL.md under 500 lines. Move detailed reference material to the `references/` directory.
