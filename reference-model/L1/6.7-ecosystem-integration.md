# Plektis Reference Model: Level 1 - Section 6.7

**Document type:** Level 1 Specification (Section 6.7)
**Version:** 0.1.0
**Date:** 2026-06-02
**Author:** Krish Jogia
**Status:** First draft. Defines how Plektis integrates the evolving agentic ecosystem and stays current without destabilising the core. Introduces the Integration Profile and the standalone Integration Profile Registry.
**Parent document:** See L0: The Node (../L0-the-node.md)

## 1. Purpose

The agentic AI ecosystem changes faster than any specification can. Hosted runtimes, tool standards, multi-agent protocols, and new model capabilities ship on a scale of weeks. A protocol that must be re-authored every time the field moves is not durable. This section defines how Plektis absorbs that change as a designed property rather than a recurring scramble.

The principle is a stable core with an extensible surface. The core of Plektis - the node, the SenRef cycle, the escalation model, and the skill file format - is deliberately small and slow-moving. The places where Plektis meets the outside world - runtimes, tools, encodings, and coordination protocols - are expected to change, and are handled through additive **Integration Profiles** rather than edits to the core.

A note on where Plektis's durable value sits. The runtime and harness layer is commoditising: the major providers have converged on near-identical managed runtimes within weeks of each other. Plektis does not compete at that layer, and it does not claim to be the interoperability layer above it - the infrastructure providers contest that ground themselves. Plektis's durable contribution is the professional-practice layer: the driver, the domain expertise encoded in skill files, and human-AI teaming as a discipline - including escalation as professional judgement and accountability, not merely a technical interrupt. The mechanisms below keep that layer portable across whatever the ecosystem produces.

## 2. Impact Taxonomy: Where a Breakthrough Lands

When a new capability or standard appears, the first question is where it touches Plektis. Five **edges** are expected to evolve and are designed to flex. One **core** is protected and changes only deliberately.

| Locus | What it covers | Example |
|---|---|---|
| **Substrate / runtime** (edge) | Where and how the machine runs | Managed agent runtimes (Anthropic Managed Agents, AWS Bedrock AgentCore, Vertex Agent Engine) |
| **Connectivity / tools** (edge) | How the machine reaches external systems | MCP, A2A, new tool or gateway standards |
| **Cognition** (edge) | How the machine reasons, remembers, and sustains autonomy | New reasoning paradigms, long-horizon autonomy, memory architectures |
| **Knowledge encoding** (edge) | How expertise is captured and loaded | Skill-file format evolutions, AAIF standard changes |
| **Interaction / teaming** (edge) | How driver and machine coordinate | New human-in-the-loop primitives, multi-agent coordination standards |
| **Charter / premise** (core) | What the node fundamentally is | A shift in autonomy or accountability that changes the human role itself |

The first five are integration surfaces. The last is the charter, and a breakthrough reaches it only when it changes what a node is.

## 3. Integration Verdict: How to Bolt It On

Each breakthrough is classified into one of four responses, in increasing order of cost and ceremony.

| Verdict | When | Mechanism | Versioning |
|---|---|---|---|
| **Watch** | Relevant but not yet actionable, or too early to commit to | A tracked entry in the registry; no spec change | None |
| **Complement** | A new place to run, connect, or encode that Plektis already supports through its open primitives | An additive Integration Profile; the core is untouched | Patch / minor |
| **Extend** | A capability the protocol cannot yet express and should | A new subsection or capability slot in the relevant section | Minor |
| **Revise** | The breakthrough invalidates a core assumption | A proposal and charter review before any change; the only path that touches L0 | Major |

One-line triage: *Does it change what the node is? Revise. A new place to run or connect? Complement. A capability the protocol cannot yet express? Extend. Otherwise, Watch.*

Most breakthroughs are Complements. Revisions are rare and deliberate.

## 4. The Integration Profile

An Integration Profile is the standard, additive artifact that maps an external system onto Plektis primitives. A profile never changes the core. Adding one is always safe. Each profile records:

- **System and verdict** - what is being integrated, its verdict, and its maturity (preview, beta, or general availability).
- **Primitive mapping** - how the external system provides or maps to each Plektis primitive:
  - **Machine and skills** - how Plektis skill files and the `AGENTS.md` bootloader load (Section 6.4).
  - **Environment and toolchain** - how the sandbox and the toolchain capability slots are provisioned (Section 6.5), including MCP.
  - **Session and persistence** - how engagement state maps to the persistence spectrum (L0 Section 3.3.5).
  - **Escalation and human-in-the-loop** - how the system's pause, approve, steer, or interrupt controls carry Plektis escalation triggers (L0 Section 3.3.4). The runtime supplies the mechanism to pause and approve. The driver and the protocol supply the judgement about when intervention is required and what the driver remains accountable for.
  - **Multi-agent and coordination** - support for the coordination grammar (Section 6.6) and inter-agent protocols such as A2A.
- **Caveats** - maturity, data-residency, regulatory, or compatibility limits.
- **Sources** - the primary documentation the profile was verified against, and the date.

Profiles live in a standalone **Integration Profile Registry** (the `integrations/` directory at the repository root), so the registry can grow, be community-contributed, and become a marketplace surface (Layer 3) without altering the reference model. Each profile describes the Plektis side once and records how the external system satisfies it, which keeps the mapping platform-agnostic.

## 5. Governance Cadence

Keeping Plektis current is a continuous practice, not an event, and it reuses a mechanism the protocol already has. Just as skill files improve through the Reflect phase of the SenRef cycle (L0 Section 3.3.6; Section 6.4), the protocol's integration surface evolves through the same loop:

1. **Scan** - track relevant breakthroughs as Watch entries.
2. **Triage** - apply the verdict rubric (Section 3).
3. **Integrate** - add or update an Integration Profile (Complement), draft a new subsection (Extend), or open a charter-level proposal (Revise).
4. **Version** - bump per the verdict, and keep the registry index and the status indexes current.

This makes evergreen behaviour a documented property of the protocol rather than a reaction to each release.

## 6. Relationship to the Core

This section is itself an edge mechanism: it can grow without touching L0. The charter, the node, SenRef, the escalation model, and the skill file format remain the stable centre. The Integration Profiles and this cadence are how that centre stays useful as the ecosystem around it changes.
