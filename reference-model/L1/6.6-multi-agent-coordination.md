## 1. Purpose

This section specifies the Multi-Agent Coordination Grammar: how multiple AI agents operating within a single node coordinate their work. Without structure, multi-agent configurations risk duplicated effort, conflicting outputs, and context loss across sessions. This section defines the rules for inter-agent communication, state persistence, and quality validation.

---

## 2. Threshold for Formal Coordination

The coordination artifacts defined here (WORKBOARD.md and HANDOFF_LOG.md) carry overhead. They cost tokens to read and time to write. They are not always required.

- **Ephemeral configurations:** Single-session multi-agent work (e.g., one agent researches while another drafts) does not require formal coordination artifacts.
- **Persistent configurations:** If work spans multiple sessions, involves persistent state, or produces artifacts that feed into client deliverables, formal coordination artifacts are required. Without them, agents lose track of what the other has done.

---

## 3. The Architect-Implementer Pattern

This is the primary coordination pattern in Plektis, tested across multiple real engagements.

- **The Architect:** Responsible for design decisions, quality standards, and review. High context, low manual execution. Interprets the driver's intent and translates it into specifications.
- **The Implementer:** Responsible for execution against the architect's specification. High throughput, focused scope. Does not make architectural decisions independently.
- **Execution Flow:** 
  1. The Driver instructs the Architect based on engagement needs.
  2. The Architect structures an implementation plan and delegates it to the Implementer via explicit schemas (see Section 6.3 for Handoff payloads).
  3. The Implementer performs the granular file generation and commits the work.
  4. The Architect executes a `/review` cycle, requesting fixes if necessary.
  5. The Architect runs `/verify` and reports systematic completion back to the Driver.

## 4. Coordination Artifacts

When formal coordination is required, these two markdown artifacts are initialised and maintained in the project's working directory. They are the single source of truth for what's been done, what's in progress, and what's next.

### 4.1 WORKBOARD.md

A kanban-style task tracker that prevents duplicate work and makes task ownership explicit.

- **Required Columns:** Backlog, Ready, In Progress, Review, Done.
- **Required Fields (per tracked item):**
  - **ID:** Standardized ticket or alphanumeric item code.
  - **Description:** Clear summary of what is being built or extracted.
  - **Owner:** Which specific agent holds the token, actively working on it.
  - **Status:** Current operational state matching the columns.
  - **Dependencies:** Explicit blockers or prerequisite items.
  - **Spec Reference:** Path pointing to the detailed architectural blueprint (`.md`).

*Rules:* An implementer may only pull items marked "Ready." Only the architect may move items to "Done" after `/verify`.

### 4.2 HANDOFF_LOG.md

An asynchronous communication ledger between agents working sequentially or in parallel.

- **Required Fields (per log entry):**
  - **Date/Timestamp.**
  - **From:** Originating agent.
  - **To:** Target agent.
  - **Task Reference:** Correlating ID linking to the WORKBOARD.
  - **Delivered:** Summary of exactly what was completed.
  - **Decided:** Any explicit logic choices or judgment calls formulated during execution.
  - **Flagged:** Pending escalations, questions, or identified blockers.

*Critical Reading Rule:* Agents must **only read entries relevant to their active task** from the log upon their cycle initialization. In a long-running engagement, a log will accumulate dozens of irrelevant historical entries - agents must filter and avoid loading stale context. Conflicting or irresolvable entries always trigger an immediate Tier 1 escalation to the Driver.

Cross-reference Section 6.3 to see canonical examples of structured information flow through these ledgers.

## 5. Alternative Topological Patterns

The Architect-Implementer pattern is the tested reference configuration. The coordination grammar also supports alternative topologies for different engagement requirements.

### 5.1 Peer Agents

**Pattern Definition:** Two or more agents possessing equal baseline authority collaborating on distinct but heavily intersecting domains.
**Example:** An M&A Financial Modeling Agent integrating simultaneously with a Supply Chain Logistics Agent.
**Implementation Mechanics:** 
- Requires dense, continuous synchronization. 
- Peer topologies *must* rely on frequent `HANDOFF_LOG.md` writes. 
- Every major structural shift must be documented to prevent architectural drift where Agent A begins using an outdated assumption invalidated by Agent B. 
- Peer agents utilize the `/verify` cycle collaboratively, rather than hierarchically. 

### 5.2 Specialist Pipeline (Linear Topology)

**Pattern Definition:** Three or more agents, each configured exclusively to operate within a single bounded SenRef cognitive layer. 
**Example:** An ingestion agent that only builds contexts (Sense Agent), passing to a structurer (Model Agent), passing to a scorer (Decide Agent).
**Implementation Mechanics:** 
- Coordination passes uniformly forward in a deterministic chain. 
- `HANDOFF_LOG.md` is linear. The Sense Agent writes its output and signs off. The Model Agent reads only the latest Sense delivery and activates. 
- Recurrent looping in pure pipeline topologies is generally an anti-pattern. If a Decide Agent identifies a data gap, it must escalate to a central Architect or Driver rather than querying the Sense Agent in reverse to preserve pipeline stability. 

### 5.3 Swarm Patterns

**Pattern Definition:** `N` agents executing highly parallel tasks simultaneously against a central generalized index or schema. 
**Example:** Processing 50 discrete target companies simultaneously to establish high-level capability profiles before consolidating them into a unified market view.
**Implementation Mechanics:** 
- WORKBOARD.md ID lockouts are essential to prevent race conditions. Each agent claims its target item ID on the board before starting work.
- Outputs are sandboxed (each agent writes to its own file, e.g., `target-alpha.md`, `target-beta.md`) before a consolidation agent merges results.
- Direct inter-agent communication via HANDOFF_LOG.md is minimised to avoid token inflation.

These topologies are configurations of the same coordination grammar. The driver selects the appropriate topology based on the engagement's requirements.

## 6. Framework Alignment

The Plektis coordination grammar aligns with patterns emerging across the major agentic AI frameworks, though it does not depend on any specific framework for implementation:

- **Google ADK** provides native `SequentialAgent` and `ParallelAgent` primitives that map to the Pipeline and Swarm topologies described above.
- **CrewAI** defines agents by role, goal, and backstory, with handoffs occurring via task dependencies. This closely mirrors the Plektis skill file approach where agents are configured through loaded context.
- **LangGraph** uses database checkpointers tied to thread IDs for state persistence across sessions, validating the same persistence imperative that drives the HANDOFF_LOG.md and WORKBOARD.md artifacts.
- **OpenAI Agents SDK** (the production successor to Swarm) implements explicit handoff primitives and I/O guardrails.
- **Microsoft Agent Framework** (merging AutoGen and Semantic Kernel) preserves conversational handoffs with added enterprise telemetry.

Plektis is framework-agnostic by design. The coordination grammar can be implemented on top of any of these frameworks, or without any framework at all (using `.md` coordination artifacts directly, as demonstrated in this project's own development process).
