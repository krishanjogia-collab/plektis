## 1. Purpose

This section defines the Machine Capability Model: what the AI agent ensemble within the node can do, organized by the SenRef cognitive cycle. For each phase, it specifies the functions the machine performs, the toolchain capability slots required (from Section 6.5), the quality criteria the driver uses to evaluate output, and the conditions that trigger escalation.

The capabilities described here are what the machine CAN do. What it SHOULD do in any given engagement is determined by the loaded skill files and the driver's mandate. The Machine Capability Model is the ceiling. The skill files are the configuration.

---

## 2. Sense Phase: Ingestion and Awareness

The machine ingests information from the external environment and the node's working memory. Raw data becomes structured awareness.

### 2.1 Functions

| Function | What the Machine Does | Example from Practice |
|---|---|---|
| **Data ingestion** | Processes financial statements, organizational documents, market reports, regulatory filings. Transforms unstructured content into structured, queryable information. | Operating model assessment: ingesting a client's organisational charts, process documentation, and system inventories to build a current-state capability profile. |
| **Multi-source monitoring** | Simultaneously tracks market signals, news feeds, regulatory changes, competitive moves. Detects patterns and anomalies across sources. | Operating model monitoring: tracking capability maturity trends across quarterly assessments to validate whether transformation investments are producing the expected improvements. |
| **Signal detection** | Identifies indicators matching predefined taxonomies - capability maturity thresholds, strategic alignment gaps, competitive shifts, operational risk patterns. | Capability assessment: detecting that three L2 capabilities scored below maturity level 2 while carrying strategic importance level 4 - a critical gap pattern requiring immediate attention. |
| **Context loading** | Reads and integrates skill files, engagement history, living artifacts, and knowledge graphs at session start. Reconstructs the node's state from persistent working memory. | Any engagement: loading the relevant domain intelligence skill file + the engagement's context document + the client's capability map from the previous session. |

### 2.2 Toolchain Slots Required

| Slot | When Required |
|---|---|
| Reasoning Engine | Always - the core processor for all Sense functions |
| Search & Grounding (via MCP) | When the engagement requires live external data (market intelligence, entity enrichment, news monitoring) |
| Document Processing (via MCP) | When ingesting client-provided documents (PDFs, Excel files, scanned images) |

### 2.3 Quality Criteria

The driver evaluates Sense-phase output against two dimensions:

**Completeness:** Did the machine process all provided input without truncation, timeout, or silent omission? A common failure mode is the machine claiming to have "reviewed" a 200-page document when it actually processed the first 50 pages. The driver should verify coverage by asking the machine to reference specific content from late in the document.

**Contextual integrity:** Does the machine correctly distinguish between immutable rules (from skill files and the protocol) and mutable data (from the client and external sources)? A common failure mode is the machine treating a client's self-reported revenue figure with the same confidence as a scoring rubric from a skill file. The driver should check whether the machine has flagged data provenance and confidence levels on ingested information.

### 2.4 Escalation Conditions

- **Tier 2 (Advisory):** The machine detects contradictions in the source data (e.g., the client's financial statements show $10M revenue but the corporate registry lists 5 employees - inconsistent for most industries).
- **Tier 2 (Advisory):** Required context files declared in a skill file's `plektis-requires` field are missing or malformed.

## 3. Model Phase: Structuring and Synthesis

The machine organizes, structures, and simulates. Raw awareness becomes usable intelligence.

### 3.1 Functions

| Function | What the Machine Does | Example from Practice |
|---|---|---|
| **Knowledge graph construction** | Builds and maintains structured representations of entities and relationships - market maps, capability taxonomies, organizational structures, portfolio configurations. | Capability modelling: constructing a capability hierarchy from L1 through L4 with PPIT dimensional mappings, using graph analysis to identify the most connected and strategically central capabilities. |
| **Scenario simulation** | Models alternative futures, portfolio configurations, deal outcomes, strategic pathways. Runs sensitivity analyses and stress tests across multiple variables. | Transformation planning: modelling three investment scenarios for a capability domain - "maintain current state," "targeted improvement," and "full redesign" - with effort estimates, timeline projections, and expected maturity outcomes for each. |
| **Pattern synthesis** | Connects signals from the Sense phase into coherent narratives - market trends, competitive dynamics, capability gaps, value chain dependencies. | Transformation planning: synthesising a capability's low maturity score, high strategic importance, understaffed PPIT profile, and aging technology stack into a coherent "priority investment" narrative with supporting evidence from the assessment data. |
| **Architecture generation** | Builds capability maps, value stream models, operating model blueprints, integration architectures. Rapid prototyping of structural designs. | Architecture visualisation: generating a Sankey diagram showing strategy-to-capability alignment, or a radar chart showing PPIT dimensional scores for an L2 capability. |

### 3.2 Toolchain Slots Required

| Slot | When Required |
|---|---|
| Reasoning Engine | Always - entity categorization, relationship identification, narrative construction |
| Structured Data Store (via MCP) | When the engagement requires persistent relational data (capability hierarchies, portfolio configurations, entity-relationship graphs) |
| Code Execution | When generating visualizations, interactive prototypes, or computed models |

### 3.3 Quality Criteria

**Structural accuracy:** Are the generated structures syntactically valid and internally consistent? A capability map with a child capability assigned to two different parents is structurally broken. A Mermaid diagram that doesn't render is syntactically invalid. The driver should spot-check structural integrity before building analysis on top of it.

**Semantic alignment:** Are entities categorized correctly within their domain hierarchy? A common failure mode is the machine placing a capability under the wrong L1 branch because the name is ambiguous (e.g., "Customer Data Management" could sit under Sales, Marketing, or IT depending on the organization's operating model). The driver should verify classification decisions against the loaded domain intelligence skill file.

**Scenario plausibility:** Do the modelled scenarios represent genuinely different futures, or are they minor variations of the same outcome? The driver should check that at least one scenario stress-tests the working thesis rather than merely confirming it.

### 3.4 Escalation Conditions

- **Tier 2 (Advisory):** The model produces results that conflict with the working thesis or reveal dynamics the driver didn't anticipate. The machine should present the conflict, not suppress it.
- **Tier 2 (Advisory):** An entity cannot be reconciled into the existing schema (e.g., a target company operates across two industries in a way that breaks the sector classification system).

## 4. Decide Phase: Scoring and Recommendation

The machine narrows options toward action. Escalation triggers are most active in this phase.

### 4.1 Functions

| Function | What the Machine Does | Example from Practice |
|---|---|---|
| **Scoring and qualification** | Applies decision frameworks to structured options - deal qualification, capability maturity assessments, market attractiveness rankings. | Capability assessment: computing maturity scores across four dimensions (maturity, strategic importance, digital readiness, sustainability) with gap deltas between current and target state, ranked by investment priority. |
| **Prioritization** | Orders opportunities, risks, or tasks by impact, urgency, feasibility, or strategic fit. Produces ranked lists the driver can act on. | Roadmap prioritisation: ranking 15 capability improvement initiatives by strategic impact score, filtering for dependency readiness and resource availability. |
| **Recommendation generation** | Synthesizes analysis into actionable recommendations with supporting evidence, confidence levels, and identified risks. The machine proposes. The driver disposes. | Capability assessment: recommending three capability investment priorities for the board, each with a FROM/TO maturity delta, estimated effort, and strategic rationale. |
| **Draft production** | Generates engagement deliverables - architecture reports, board presentations, assessment summaries, transformation roadmaps, investment cases. Produces complete first drafts that the driver refines. | Strategy execution: drafting a board-ready transformation business case with executive summary, capability gap analysis, investment recommendations, and expected outcomes. |

### 4.2 Toolchain Slots Required

| Slot | When Required |
|---|---|
| Reasoning Engine | Always - the core processor for judgment, scoring, and narrative synthesis |
| Implementation Engine | When the engagement requires high-volume document production or complex formatting |

### 4.3 Quality Criteria

**Logical consistency:** Does the scoring output correlate to the defined rubric? If the skill file defines a scoring band of 0-30 as "Low" and the machine scores an entity at 25 but labels it "Medium," the output is logically inconsistent. The driver should verify that scores, labels, and rankings all align with the loaded scoring methodology.

**Tone and format alignment:** Do drafted deliverables sound like they were written by a senior practitioner, not a chatbot? The test: would a partner sign this as their own work? A common failure mode is the machine producing correct content in a generic, AI-sounding register. The loaded domain intelligence skill file should calibrate tone - the driver verifies it was applied.

**Confidence honesty:** Does the machine state its confidence accurately? Over-confidence on speculative analysis is more dangerous than under-confidence on solid data. The driver should check that low-confidence outputs are explicitly flagged, not buried in confident-sounding prose.

### 4.4 Escalation Conditions

- **Tier 1 (Mandatory):** Any recommendation involving capital commitment, legal obligation, or irreversible strategic action (asset sale, market exit, organizational restructuring).
- **Tier 1 (Mandatory):** Any draft communication intended for external parties (clients, counterparties, regulators, boards).
- **Tier 1 (Mandatory):** Output that deviates significantly from the driver's initial constraint parameters.
- **Tier 2 (Advisory):** Multiple valid options exist where the choice depends on values, risk appetite, or political considerations the machine cannot assess.

## 5. Reflect Phase: Persistence and Learning

The machine evaluates outcomes, updates assumptions, and refines its own capabilities. This is how the node gets smarter over time.

### 5.1 Functions

| Function | What the Machine Does | Example from Practice |
|---|---|---|
| **State serialization** | Persists ephemeral working state to durable storage - writing updated context files, saving graph state, committing engagement artifacts. | Any engagement: writing the updated engagement context document, saving the modified capability scores to the database, committing the session's work to version control. |
| **Handoff documentation** | Produces structured records of what was done, decided, and flagged - HANDOFF_LOG.md entries, session summaries, delivery notes. | Multi-agent work: writing a handoff entry documenting what was built, what decisions were made, what remains open, and what the driver should review. |
| **Skill file refinement** | Proposes updates to skill files based on engagement experience - new anti-patterns discovered, scoring weights that need adjustment, methodology steps that should be reordered. | Capability assessment: proposing a new scoring criterion after the driver corrected the maturity assessment approach three times across engagements. The correction becomes a formal rule in the skill file. |
| **Meta-analysis** | Evaluates the node's own performance - where did the SenRef cycle produce strong output, where did it break down, which phases needed more driver oversight. | Post-engagement: noting that the Sense phase missed a critical signal because the search grounding MCP server didn't cover industry-specific databases. Flagging this as a toolchain gap. |

### 5.2 Toolchain Slots Required

| Slot | When Required |
|---|---|
| Persistent Context Store | Always - writing state to `.md` files or equivalent |
| Version Control (via MCP) | When the engagement requires auditable provenance (commit history) |
| Reasoning Engine | For self-evaluation and skill file refinement proposals |

### 5.3 Quality Criteria

**State fidelity:** Does the persisted state accurately reflect what happened during the session? A common failure mode is the machine summarizing rather than preserving - writing "discussed three scenarios" instead of persisting the actual three scenarios with their parameters and outcomes. The driver should verify that persisted artifacts are complete enough to reconstruct the session's work.

**Metadata hygiene:** Are version numbers, dates, author fields, and provenance headers correct? Sloppy metadata degrades the institutional knowledge base over time. The driver should spot-check metadata on any artifact that enters the cross-engagement persistent layer (skill files, reference architectures).

**Proposal specificity:** When proposing skill file updates, does the machine identify the specific section, the specific change, and the specific engagement evidence that justifies it? "The scoring weights should be adjusted" is not a proposal. "Section 2.1, Maturity Level 3 criteria, add 'documented process ownership' as a requirement, based on three engagements where capabilities scored at Level 3 without documented ownership consistently failed to maintain that maturity" is a proposal.

### 5.4 Escalation Conditions

- **Tier 1 (Mandatory):** Any persistence action that would overwrite a critical preexisting document or dataset. Overwriting is irreversible without version control. The driver must authorize destructive persistence actions.
- **Tier 3 (Informational):** Skill file refinement proposals. The driver reviews and approves before they become canonical.

## 6. Cross-Phase Capabilities

These capabilities span the full SenRef cycle and are not bound to a single phase.

### 6.1 Persistent Context Maintenance

The machine holds and navigates the full state of an engagement - every loaded skill file, every conversation exchange, every data point - without cognitive fatigue. This enables the machine to reference a detail from page 47 of a capability assessment report while drafting a board presentation, or to connect a signal detected in Sense to a recommendation produced in Decide.

**Quality test:** Ask the machine to reference specific details from early in the engagement. If it cannot, context has been lost or truncated.

### 6.2 Confidence Communication

The machine must proactively communicate its confidence level on every substantive output. This is not optional - it is an architectural requirement (L0 Operating Principle #4). High confidence on data synthesis from verified sources. Medium confidence on pattern inference from partial data. Low confidence on strategic speculation.

**Quality test:** Check that confidence signals vary across outputs. If every output is "high confidence," the machine is not calibrating honestly.

### 6.3 Format Transformation

The machine translates between formats fluidly - markdown documentation to JSON data structures, analysis narratives to Mermaid diagrams, financial models to board-ready summaries. The same underlying intelligence is expressed differently for different consumers.

**Quality test:** The transformation should preserve the substance while changing the container. If a board summary omits a critical risk that was in the analysis narrative, the transformation failed.

### 6.4 Multi-Agent Coordination Readiness

When operating in a multi-agent configuration (Section 6.6), the machine must read and respect coordination artifacts (WORKBOARD.md, HANDOFF_LOG.md), wait for peer agents rather than duplicating their work, and clearly delineate its own contributions from others'.

**Quality test:** In a multi-agent engagement, can the driver trace which agent produced which output? If attribution is unclear, coordination has failed.

## 7. Capability Evolution

Machine capabilities are not static. As AI models improve, the boundary between what requires driver intervention and what the machine can handle autonomously shifts. The protocol accommodates this through two mechanisms:

1. **Escalation threshold adjustment.** A function that required Tier 1 escalation in 2026 (e.g., generating a first-draft board presentation) may require only Tier 2 or Tier 3 as models become more reliable. The driver adjusts escalation profiles (Section 6.5, Section 2.4) based on observed machine performance in their specific domain.

2. **Skill file evolution.** As machine capabilities grow, skill files evolve to delegate more work to the machine and reserve less for the driver. The anti-patterns section shrinks (fewer failure modes) while the workflow steps expand (more complex delegated tasks). The evolution protocol (L0 Section 3.3.6, Skill File Architecture Section 6.4) governs this progression.

The Machine Capability Model described in this section represents the current state of the art. It will be revised as the underlying models evolve.
