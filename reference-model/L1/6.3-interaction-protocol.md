## 1. Purpose

This section defines the Interaction Protocol Specification: the structure of handoffs, escalation routes, and feedback loops between actors in the node. When assumptions stay unspoken, nodes fail. These protocols make assumptions explicit.

**Prompts vs. protocols:** This specification defines *what information must be exchanged*, not *how to format a prompt*. Prompt formatting is a toolchain-specific detail. The protocol is platform-agnostic.

---

## 2. Interaction Handoff Specifications

The reference model defines the schema for what handoffs must contain. The SDK (Layer 2) provides downloadable templates. Any interaction that crosses a boundary within the node should include these fields.

### 2.1 Driver → Machine Handoff

When the driver delegates work to the machine, the handoff should contain these fields. Missing fields lead to misaligned output.

- **Objective/Mandate:** What exactly needs to be achieved in this cycle.
- **Scope Limits:** Strict boundaries of what the machine must NOT touch or output.
- **Constraints/Rules:** Stylistic, logic, or format directives.
- **Quality Criteria:** What success looks like from the driver's perspective.
- **Escalation Triggers Active:** Which specific Tier 1 or Tier 2 limits are exceptionally relevant to this task.
- **Context Files Loaded:** A manifest of which `.md` playbooks or intelligence schemas have been primed in the machine's context window.
- **Confidence Threshold Required:** Usually, defining how certain the machine must be before acting versus escalating.

**Reference Format with Realistic Payload:**

```markdown
## Handoff: Driver → Machine

**Objective:** Map the acquired target's supply chain redundancies into a standard Capability Model (Level 2).
**Scope:** Do NOT assess financial integrations. Confine mapping exclusively to operational logistics and warehouse capabilities. 
**Constraints:** Output must be valid Mermaid flowchart syntax (TD workflow). Use standard `SKILL.md` nomenclature for nodes. No emoji mapping.
**Quality criteria:** The flow must demonstrate clearly where raw material inbound paths overlap between our baseline and the target.
**Active escalation triggers:** [Tier 1] If source documents conflict regarding existing 3PL contracts, STOP and escalate to Driver. Do not hallucinate a reconciliation.
**Context loaded:** 
  - `playbooks/capability-assessment.md`
  - `data/target-ops-summary.pdf`
**Confidence threshold:** High. If inference is required to bridge logistics nodes, escalate instead.
```

### 2.2 Machine → Driver Handoff

When the machine completes a SenRef cycle and returns control, it reports what happened. The driver needs to understand what changed in the engagement state.

- **Delivery Summary:** What was accomplished.
- **Omissions:** What was intentionally skipped or failed.
- **Confidence Assessment:** An explicit label (e.g., High, Medium, Low) on the fidelity of the output.
- **Escalation Flags:** Any Tier 1 or Tier 2 triggers tripped during execution.
- **Anomalies Detected:** Data or logic mismatches found in the input context.
- **Persistence Actions Taken:** What was written to working memory during this cycle. The driver must know what changed in the engagement state. 
- **Open Questions:** Where the driver needs to provide further resolution.

**Reference Format with Realistic Payload:**

```markdown
## Handoff: Machine → Driver

**Delivered:** Transformed target-ops-summary.pdf into `target-supply-chain-L2.mmd`. Identified 14 distinct logistics capabilities.
**Omitted:** Excluded the "freight auditing" node; documentation was insufficient to establish its placement in the hierarchy.
**Confidence:** Medium. Some capabilities were heavily abstracted in the source PDF, requiring standardized estimations.
**Escalation flags:** NONE. Source docs were cleanly aligned on 3PL contracts.
**Anomalies:** The `baseline` supply chain structure you uploaded last week uses a 4-tier model, whereas the target utilizes a flat organizational topology. Direct 1:1 overlap mapping might be difficult.
**Persistence actions:**
  - Generated new file: `target-supply-chain-L2.mmd`
  - Wrote status update to `engagement-status-tracker.md` to reflect task completion.
**Open questions:** Should I proceed with attempting to normalize the target's flat topology into our 4-tier model, or preserve its native hierarchy for the next review?
```

### 2.3 Machine → Machine Handoff (Multi-Agent)

In multi-agent configurations using the Architect-Implementer pattern (Section 6.6), the architect agent delivers explicit handoffs to the implementer agent.

- **Task ID / Reference:** To track items.
- **Design Spec Reference:** Pointer to the exact architectural ruling.
- **Acceptance Criteria:** Definition of done.
- **Conventions:** Stylistic standards required.
- **Review Expectations:** Whether a `/review` cycle is blocking the next step.

*Note:* This explicitly references the `WORKBOARD.md` and `HANDOFF_LOG.md` artifacts detailed in L0 Section 3.3.3.

**Reference Format with Realistic Payload:**

```markdown
## Handoff: Architect → Implementer

**Task ID:** TASK-412 (Generate Integration Node Visualizations)
**Design spec:** `specs/integration-reporting-standards.md` (Node boundaries section)
**Acceptance criteria:** Create three Mermaid graphs (Current State, Transition State, Future State). All graphs must compile without syntax errors natively.
**Conventions:** Use `:::focus` class tags on nodes that represent new capability injections. Prefix all node IDs with `int_`.
**Review expectation:** Block. Submit initial schemas to me for `/review` before finalizing them into the core repository docs.
```

## 3. Quality Gate Protocols

Structured quality gates prevent output quality from degrading over long engagements.

### 3.1 The `/review` Cycle

The review cycle checks whether an implementer's delivery matches the architect's design intent.

- **Trigger:** After an implementation delivery is returned to the architect.
- **Checklist:**
  1. *Design intent accuracy:* Does this functionally solve the mandate?
  2. *Architectural consistency:* Does this break established schema or reference existing files incorrectly?
  3. *Convention compliance:* Have styling, markdown norms, and specific operational rules been upheld?
  4. *Content preservation:* Were unrelated sections of code or comments inadvertently wiped or overwritten?
- **Output:** A prioritised fix list with specific files and locations.
- **Re-review:** The cycle repeats until the fix list is clear.

### 3.2 The `/verify` Cycle

Review checks the work just completed (atomic). Verify checks the integrated system (systemic). They are distinct gates.

- **Trigger:** When the architect needs to confirm the overall system is valid, typically before delivering to the driver.
- **Checklist:**
  1. *Functional correctness:* Does the integrated logic hold? Does the system parse or compile?
  2. *Completeness:* Were all elements of the driver's mandate addressed?
  3. *Regression:* Did new work break something established earlier in the engagement?
- **Output:** Go/No-Go. A No-Go on systemic issues escalates to the driver.
