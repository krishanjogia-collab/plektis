## 1. Purpose

This section defines the Driver Capability Model: what the human operator needs to know and do, organised by competency domain. The node's effectiveness is constrained by the competence of the human operating it. The machine executes within boundaries the driver sets. If those boundaries are poorly defined, the output is technically correct but strategically useless.

The driver is not a consumer of AI outputs. The driver is the orchestrating intelligence that directs the SenRef cycle, authorises consequential decisions, and navigates the social reality of the engagement that the machine cannot perceive.

## 2. The Six Competency Domains

Driver capability operates across six domains. These are areas of active judgment, not procedural checklists. They map to the driver capabilities defined in L0 Section 3.1 and are independent of which AI platform or toolchain is used.

### 2.1 Strategic Translation

**What it is:** Converting ambiguous business mandates into clear, machine-actionable context.

**Why it matters:** The machine executes within whatever constraints it receives. Poorly translated context produces outputs that follow instructions but miss the strategic point.

The driver must:
- Understand the client's political and strategic reality before briefing the machine
- Define the engagement's core objective and boundary conditions
- Curate and structure the context files the machine will use
- Articulate constraints on how the machine should approach the work (methodology, format, tone, sensitivities)

### 2.2 Escalation Judgment

**What it is:** Recognising when machine output crosses a threshold requiring human authorisation, and evaluating that output rather than rubber-stamping it.

**Why it matters:** The machine cannot absorb legal, financial, or strategic liability. It produces recommendations. The driver carries responsibility for acting on them.

The driver must:
- Operate the three-tier escalation framework (L0 Section 3.3.4)
- Assess whether Decide-phase output is trustworthy for the specific engagement context
- Validate decisions against client constraints before authorising any external action (e.g., stopping an unauthorised communication before it reaches a counterparty)

### 2.3 Stakeholder Calibration

**What it is:** Reading rooms, managing organisational dynamics, and calibrating machine-produced deliverables to match the audience.

**Why it matters:** The machine does not have access to the social dynamics of an organisation. It does not know that the board prefers concise summaries while the operating team needs detailed breakdowns. The driver bridges this gap.

The driver must:
- Determine who sees what, in what format, and at what level of detail
- Intercept outputs that are analytically sound but politically problematic
- Restructure deliverables for the specific audience without losing the substance

### 2.4 Quality Governance

**What it is:** Reviewing machine outputs against domain standards and maintaining the integrity of the node's knowledge base.

**Why it matters:** Machine output quality degrades over long engagements. Context windows fill, assumptions compound, and small errors propagate. The driver is the quality gate.

The driver must:
- Execute `/review` and `/verify` cycles to confirm outputs meet architectural intent and domain standards
- Ensure deliverables meet the quality bar of senior advisory work, not just "AI-generated content"
- Catch regressions, hallucinations, and logic drift in complex multi-session engagements

### 2.5 Methodology Selection

**What it is:** Choosing which skill files to load, which playbooks to execute, and which configuration to establish for a given engagement.

**Why it matters:** Loading the wrong skill files produces misaligned output. Running an M&A playbook during a capability assessment engagement wastes time and creates confusion. The mounting decision (Section 6.5) is a driver responsibility.

The driver must:
- Select the appropriate Domain Intelligence and Operational Playbook skill files for the engagement
- Configure the toolchain capability slots needed (Section 6.5)
- Set the persistence level (ephemeral, session, engagement, or cross-engagement)
- Connect the required MCP servers for external data access

### 2.6 Institutional Learning

**What it is:** Driving the Reflect phase across engagements by codifying corrections and evolving the methodology based on what worked and what did not.

**Why it matters:** The node must get smarter over time. The machine cannot self-improve without the driver identifying what needs to change. This is how a mature skill file evolves - from v1 to v3+ through iterative rules, each discovered during a real engagement.

The driver must:
- Post-mortem engagements to identify what succeeded and what failed
- Codify new anti-patterns, quality rules, and methodology refinements into skill files
- Version-bump updated skill files and document what changed and why

## 3. Driver Certification Readiness Criteria

Certification is a Layer 5 commercial capability, but the competency standards it tests belong in the reference model. Certification does not test knowledge of YAML syntax or markdown formatting. It tests whether a practitioner can safely and effectively operate the human side of a node.

A candidate is assessed against these readiness criteria:

1. **Translation Readiness:** Given a raw, unstructured client brief (a transcript, a scattered email chain, a set of financials), can the candidate produce a machine-actionable handoff that successfully initialises a SenRef cycle?

2. **Escalation Security:** Given a simulated machine output with seeded anomalies (a recommendation that violates a stated constraint, a score that contradicts the underlying data), does the candidate catch them and halt execution?

3. **Stakeholder Calibration:** Given a technically correct but tonally inappropriate machine-drafted deliverable, can the candidate restructure it for a specific audience without losing the analytical substance?

4. **Toolchain Competence:** Given an engagement scenario, can the candidate select the correct skill files, configure the appropriate capability slots (Section 6.5), and connect the right MCP servers?

5. **Quality Review:** Can the candidate execute a `/review` cycle that produces a specific, actionable fix list (not vague feedback like "make it better")?

6. **Institutional Learning:** Does the candidate habitually persist session state, update context files, and propose skill file refinements based on engagement experience?

**Note on assessment format:** This section defines the competencies a driver must demonstrate. It does not prescribe how a certification program tests them. The proven scaling model for professional certifications (SAFe, Scrum, TOGAF, CBA) is scenario-based assessment via proctored platforms, not live practical simulations. Certification programs built on Plektis should design assessment items that test these competencies within a format that scales globally. The readiness criteria above describe what to test. The certification program determines how.
