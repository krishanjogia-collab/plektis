## 1. Purpose

This section defines the Application Mounting Protocol: how domain-specific applications connect to the protocol and access its services. "Mounting" means configuring the node to execute a specific type of work - loading the right skill files, provisioning the right tools, establishing the right persistence, and initializing the SenRef cycle for that domain.

The protocol is a human-AI teaming standard. Applications are what practitioners actually do with it. Mounting is the bridge between the two.

This section answers three questions:

1. **What does an application need from the protocol?** (The Mounting Interface)
2. **What tools does the node need, and how are they provisioned?** (The Toolchain Capability Map)
3. **How does a specific domain application configure the node?** (The Mounting Sequence)

## 2. The Mounting Interface

An application that mounts onto the protocol must provide four things:

### 2.1 Domain Intelligence

One or more domain intelligence skill files that configure the machine's knowledge for the engagement domain. These are loaded as persistent context and calibrate the SenRef cycle for the specific domain.

**Examples:**

| Application Domain | Domain Intelligence Skill(s) |
|---|---|
| Capability assessment | `business-architecture-foundations` (capability mapping, PPIT dimensions, maturity scoring) |
| Operating model design | `business-architecture-foundations` + sector-specific domain skill |
| Strategy-to-execution | `business-architecture-foundations` (BMM traceability, initiative mapping) |

An application may load multiple domain intelligence skills simultaneously. The skill file's `plektis-requires` metadata field declares dependencies between skills.

### 2.2 Operational Playbooks

One or more operational playbook skill files that define the workflows the machine executes. These prescribe the SenRef cycle's behavior for specific tasks within the domain.

**Examples:**

| Application Domain | Operational Playbook(s) |
|---|---|
| Full capability assessment | Assessment workflow playbook + gap analysis playbook |
| Target operating model design | TOM design playbook (PPIT dimensional analysis) |
| Strategy-to-execution | Strategy mapping playbook + initiative prioritisation playbook |

An application may have no playbooks (if the domain intelligence alone is sufficient to calibrate the machine) or many (if the engagement involves multiple structured workflows).

### 2.3 Persistence Configuration

A declaration of how the application manages state, mapping to the persistence spectrum defined in L0 Section 3.3.5:

| Persistence Level | When to Use | Mechanism |
|---|---|---|
| Ephemeral | Single-session exploration, prototyping | In-memory (conversation context, graphology.js) |
| Session-persistent | Multi-session engagement, working continuity | Project memory, working documents |
| Engagement-persistent | Full lifecycle, auditable deliverables | `.md` files, SQLite/Prisma, serialized graph state |
| Cross-engagement persistent | Institutional knowledge, reusable methodology | Skill files, practitioner profiles, reference architectures |

Every application must declare its minimum persistence level. Layer 5 deployments require engagement-persistent or higher. Layer 2 SDK explorations may be ephemeral.

### 2.4 Escalation Profile

A declaration of how the application configures the three-tier escalation framework (L0 Section 3.3.4) for its specific domain. Different domains have different escalation sensitivities:

| Application Domain | Tier 1 (Mandatory) | Tier 2 (Advisory) |
|---|---|---|
| Strategy advisory | Any recommendation involving significant investment commitment or organisational restructuring | Findings that conflict with the strategic direction, scenarios with materially different outcomes |
| Market intelligence | Any outreach to external parties, any data that could identify individuals | Data anomalies, entity resolution conflicts, conflicting sources |
| Business architecture | Any deliverable sent to client stakeholders, any recommendation involving organizational restructuring | Capability assessment scores that conflict with stakeholder expectations |

The escalation profile is not a new framework - it configures the existing L0 escalation triggers for the domain's specific risk landscape.

## 3. The Node Toolchain Capability Map

The protocol defines *what the node needs to do*. The toolchain defines *what software provisions those capabilities*. The toolchain is pluggable - practitioners choose their own tools as long as each logical capability is fulfilled.

### 3.1 Capability Slots

Every node requires these logical capabilities. The protocol does not prescribe which vendor provides them.

| Capability Slot | What It Does | SenRef Phase | Required? |
|---|---|---|---|
| **Reasoning Engine** | Deep analysis, strategy formulation, architecture design, complex judgment | All phases | Yes |
| **Code Execution** | Build, test, deploy software artifacts | Model, Decide | Only for Layer 4 product development |
| **Implementation Engine** | High-volume production against specs (code, documents, data processing) | Model, Decide | Only for multi-agent configurations |
| **Search & Grounding** | Real-time web data, market intelligence, entity enrichment | Sense | Only for applications requiring live data |
| **API Gateway** | Model access for products (inference endpoints for Layer 4 applications) | All phases | Only for deployed products |
| **Version Control** | Source of truth, provenance, collaboration | Reflect | Yes (for any persistent engagement) |
| **Deployment Platform** | Hosting prototypes and products | N/A (infrastructure) | Only for Layer 4 products |
| **Persistent Context Store** | Cross-session knowledge, skill files, living artifacts | All phases | Yes |
| **Structured Data Store** | Entity-relationship state, capability hierarchies, financial models | Model, Decide | Only for applications with complex relational data |
| **Document Processing** | Ingest, extract, and transform client documents (PDFs, Excel, scanned images) | Sense | Only for applications that receive client documents |

### 3.2 Reference Implementation

The following is one implementation of the toolchain - the configuration used by the protocol's author. It is not prescriptive. It demonstrates that each capability slot can be filled by a specific tool.

| Capability Slot | Our Tool | Alternatives (Non-Exhaustive) |
|---|---|---|
| Reasoning Engine | Claude (chat/projects) | GPT-4o, Gemini Advanced, Llama via Ollama |
| Code Execution | Claude Code | Cursor, GitHub Copilot, Windsurf, Gemini Code Assist |
| Implementation Engine | Google Gemini | Claude Code, Cursor (same tool, different role) |
| Search & Grounding | Gemini (via API, search grounding) | Perplexity API, Tavily, Bing Search API |
| API Gateway | Google AI Studio | Anthropic Console, OpenAI Platform, AWS Bedrock |
| Version Control | GitHub | GitLab, Bitbucket |
| Deployment Platform | Vercel | Netlify, AWS Amplify, Cloudflare Pages |
| Persistent Context Store | `.md` files (local filesystem / GitHub) | Any file system, Notion, Obsidian |
| Structured Data Store | SQLite + Prisma, graphology.js | Neo4j, PostgreSQL, TigerGraph, DuckDB |
| Document Processing | (not yet implemented) | Unstructured.io, AWS Textract, LlamaIndex |

### 3.3 Vendor Independence Principle

The protocol must never couple to a specific vendor's capabilities. This means:

1. **Skill files are platform-agnostic.** They are `.md` files. They work on Claude, Gemini, GPT, or any agent that reads markdown. The AAIF conformance ensures interoperability across the ecosystem.

2. **The SenRef cycle is model-agnostic.** Sense, Model, Decide, Reflect are cognitive phases, not API calls. Any reasoning engine capable of structured analysis can execute them.

3. **Persistence is mechanism-agnostic.** The protocol requires persistence (L0 Section 3.3.5) but not a specific technology. `.md` files, databases, graph stores, and serialized state are all valid.

4. **Multi-agent coordination is framework-agnostic.** The architect-implementer pattern (L0 Section 3.3.3) works across Claude + Gemini, Claude + Claude Code, GPT + Copilot, or any two agents that can read shared `.md` coordination artifacts (WORKBOARD.md, HANDOFF_LOG.md).

5. **Integration is protocol-agnostic via MCP.** The protocol requires external connectivity for capability slots (Section 3.4) but does not mandate specific MCP server implementations. Any MCP-compliant server that provisions the required capability slot is valid. This ensures the node can integrate with any enterprise's existing tool stack.

When documenting a reference implementation (as in 3.2), always list alternatives. When writing skill files, never reference a specific vendor in the instructions - reference the capability slot instead ("use the search grounding capability" not "use Gemini search grounding").

### 3.4 Integration Protocol: MCP (Model Context Protocol)

The Toolchain Capability Map defines *what* the node needs. Skill files define *how the agent thinks*. The missing piece is *how the agent connects to live tools, data, and enterprise systems at runtime*. This is the role of the Model Context Protocol (MCP).

MCP is an AAIF standard (contributed by Anthropic, December 2025) that provides a universal, bidirectional protocol for connecting AI models to external tools, data repositories, and enterprise applications. It is the active integration layer: skill files are passive instruction documents that configure cognition. MCP is the live wiring that connects the machine to its environment.

The official MCP server registry is at `registry.modelcontextprotocol.io`. Anthropic maintains reference server implementations at `github.com/modelcontextprotocol/servers`.

**The AAIF two-pillar model:**

| Pillar | Standard | What It Does | Plektis Layer |
|---|---|---|---|
| **Cognition** | Agent Skills (SKILL.md) | Defines how the agent thinks and what it knows | Plektis skill files (Layer 2 SDK) |
| **Integration** | Model Context Protocol (MCP) | Connects the agent to tools, data, and systems | Toolchain provisioning (this section) |

Both pillars are required for a functioning node. A node with skill files but no MCP connections is a well-configured agent that cannot reach its data. A node with MCP connections but no skill files is a connected agent that lacks domain expertise. Plektis requires both.

**How MCP maps to capability slots:**

Each capability slot in the Toolchain Capability Map (Section 3.1) that involves live external access can be provisioned via an MCP server:

| Capability Slot | MCP Server Role | Ecosystem Status (as of April 2026) |
|---|---|---|
| Search & Grounding | Connects to web search, news feeds, market data | **Available.** Brave Search, Google Maps, community servers. Anthropic provides official web fetch and Puppeteer servers. |
| Structured Data Store | Connects to databases, graph stores, enterprise data | **Available.** Anthropic provides official PostgreSQL and SQLite reference servers. SAP LeanIX and Ardoq offer read-only EA data integrations. |
| Document Processing | Connects to file systems, document stores, OCR | **Partial.** Official filesystem server available. Heavy OCR and document extraction relies on community servers or adjacent API providers. |
| Version Control | Connects to git hosting | **Available.** Official GitHub and GitLab MCP servers in the reference repository. |
| Deployment Platform | Connects to CI/CD and hosting platforms | **Gap.** No production MCP servers currently exist for Vercel, Netlify, or AWS deployment. CI/CD remains handled through traditional pipeline tools outside the MCP layer. |
| API Gateway | Connects to model inference endpoints | **Sparse.** Provider-specific; not a primary MCP use case. |
| Version Control | Connects to git hosting | GitHub MCP, GitLab MCP |

Capability slots that are internal to the agent (Reasoning Engine, Persistent Context Store) do not require MCP - they are native to the AI platform.

**MCP in the mounting sequence:**

When an application mounts onto the protocol (Section 4), MCP connections are provisioned as part of Step 1 (Provision the Toolchain). The application's mounting interface (Section 2) should declare which MCP servers are required:

```yaml
# Example: Capability assessment application mount
mcp-servers:
  - type: structured-data
    server: postgresql         # or sqlite for lightweight deployments
    purpose: "Capability hierarchy and PPIT assessment data persistence"
  - type: version-control
    server: github
    purpose: "Engagement artifact versioning and provenance"
```

This declaration is informational, not enforced at the protocol level. The driver provisions the MCP servers that match their toolchain. The protocol specifies what *kind* of connectivity is needed. The implementation decides *which* MCP server provides it.

**Security implications:**

MCP connections introduce the same trust boundary concerns as skill files (see Skill File Architecture, Section 4.3). Specifically:

1. **MCP servers can access external data.** Any data ingested through an MCP connection enters the node's working memory and may influence SenRef cycle outputs. The node must treat MCP-sourced data with the same confidence-awareness required by L0 Operating Principle #4.

2. **MCP servers can execute actions.** Some MCP servers support write operations (creating files, updating databases, sending messages). These actions must respect the escalation framework - any MCP-triggered action that would constitute a Tier 1 or Tier 2 escalation must be gated by the driver.

3. **MCP server provenance matters.** Community-developed MCP servers carry the same supply-chain risk as community-sourced skill files. The trust level framework (first-party, curated, community) should extend to MCP server selection. For Layer 5 deployments, only first-party or curated MCP servers should be used.

## 4. The Mounting Sequence

When a practitioner configures the node for a specific engagement, the following sequence executes. This may be manual (the practitioner loads files and configures tools) or automated (the AGENTS.md bootloader handles initialization).

### 4.1 Step 1: Provision the Toolchain

Select and configure tools for each required capability slot. At minimum: a reasoning engine and a persistent context store. Additional slots are provisioned based on the application's requirements.

### 4.2 Step 2: Connect MCP Servers

Provision the MCP connections required by the application. Each capability slot that involves external access should be connected through a standards-compliant MCP server. Verify connectivity before proceeding - a Sense-phase skill file that expects search grounding will fail silently if the MCP server is not connected.

For prototype/exploration work, MCP connections may be stubbed or omitted. For Layer 5 deployments, all declared MCP connections must be live and tested.

### 4.3 Step 3: Load Domain Intelligence

Load the domain intelligence skill file(s) into the reasoning engine's context. These calibrate the machine's knowledge and quality standards for the engagement domain.

The skill file's `plektis-requires` metadata field declares dependencies. Load dependencies first.

### 4.4 Step 4: Load Operational Playbooks

Load the operational playbook skill file(s) into the reasoning engine's context. These define the workflows the machine will execute.

The skill file's `plektis-senref-phases` metadata field indicates which SenRef phases the playbook covers. Verify that the loaded playbooks provide coverage across the phases required for the engagement.

### 4.5 Step 5: Configure Persistence

Establish the persistence mechanism per the application's declared persistence level:

- **Ephemeral:** No setup needed. State lives in the session.
- **Session-persistent:** Create or open a project context (e.g., Claude Projects, a working directory).
- **Engagement-persistent:** Initialize the data store (create the `.md` context files, initialize the database, set up the working directory structure).
- **Cross-engagement persistent:** Verify that institutional skill files and reference architectures are available and up to date.

### 4.6 Step 6: Activate the Escalation Profile

Configure the escalation triggers for the domain. The driver reviews the application's escalation profile (Section 2.4) and confirms the Tier 1 and Tier 2 triggers are appropriate for this specific engagement.

### 4.7 Step 7: Initialize the SenRef Cycle

The node is mounted. The machine begins in the Sense phase - ingesting the engagement context, client data, and any existing artifacts. The driver provides the engagement mandate. The SenRef cycle begins.

## 5. Application Registry (Current)

Each application that mounts onto the protocol combines **public components** - a Deployment Profile and the open skills and SDK it draws on - with **commercial-layer components**: the proprietary instruments and playbooks that ship in a full Layer 4 / Layer 5 deployment rather than in the public SDK.

| Application | Domain Intelligence | Playbooks | Persistence | Toolchain Slots | MCP Connections |
|---|---|---|---|---|---|
| **Business Architecture** | Deployment Profile (public, [`configurations/`](../../configurations/business-architecture.md)) + the open Business Architecture Foundations skill | Proprietary engagement playbooks † (mapping, heat-mapping, and roadmapping workflows) | Engagement-persistent (`.md` + SQLite) | Reasoning, Code Execution, Version Control, Persistent Context, Structured Data | Structured Data (SQLite/PostgreSQL), Filesystem |
| **Portfolio and Transaction Strategy** | Deployment Profile (public, [`configurations/`](../../configurations/portfolio-and-transaction-strategy.md)) | Proprietary qualification and pipeline instruments, frameworks, playbooks, and the succession-signal taxonomy † | Engagement-persistent (`.md` + structured store) | Reasoning, Structured Data, Persistent Context | Structured Data, Filesystem |

† Commercial-layer component, available in a full Layer 4 / Layer 5 deployment rather than the public SDK. The public profile and any open skill it mounts provide the foundation. The proprietary instruments and playbooks add the engagement depth.

Business architecture feeds downstream work such as operating-model design, investment cases, and transformation planning. Those consume the architecture rather than appearing as separate entries here.

## 6. Deployment Profiles

A Deployment Profile is the public, published form of an application's mounting. It lives in `configurations/` and shows what it looks like to run a domain as a node: how an engagement maps to SenRef, where the machine stops, which capability slots it draws, what it persists, and what mounts publicly or privately.

A profile carries the shared vocabulary of a domain and the shape of the node that runs it. The scored instruments and the engagement depth stay private, in a Layer 4 or 5 deployment. The profiles so far are [Portfolio and Transaction Strategy](../../configurations/portfolio-and-transaction-strategy.md) and [Business Architecture](../../configurations/business-architecture.md).
